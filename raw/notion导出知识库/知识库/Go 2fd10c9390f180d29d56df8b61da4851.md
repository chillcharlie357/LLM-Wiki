# 基础数据结构

## **Slice（切片）**：

### 切片和数组的区别

- 切片在运行时，切片由 `reflect.SliceHeader` 定义：
    - **Data**: 指向底层数组的指针。
    - **Len**: 当前切片的长度。
    - **Cap**: 底层数组从第一个元素开始到末尾的总容量。

| 特性  | 数组 (Array) | 切片 (Slice) |
| --- | --- | --- |
| 长度 | 固定，定义后不可变 | 动态，可扩容 |
| 类型定义 | 长度是类型的一部分,如 [3]int 和 [4]int 是不同类型 | 长度不是类型的一部分，如 []int |
| 类型本质 | 值类型 (Value Type) | 引用类型 (结构体描述符) |
| 传递方式 | 函数传参拷贝整个数组 | 函数传参传递引用（指针），效率高 |
| 内存分配 | 编译时分配，空间固定 | 运行时动态分配 |
| 声明语法 | [5]int (指定长度) | []int (不指定长度) |

### `append` 的扩容机制

1.18 版本后：

- 扩容策略：
    - 旧容量 < 256 时翻倍
    - 旧容量> 256 时则使用公式 $newcap = oldcap + (oldcap + 3 \times 256) / 4$ 逐步递增，直到大于所需容量。
- 内存对齐： 计算完会进行内存对齐，最终容量会略大

- `append`可能返回新的切片引用，必须写成 `s = append(s, ...)`

### 复制

切片截取`s1 := s[1:2]`会共享底层`array`，修改和`append`时要注意:

```go
package main

import "fmt"

func main() {
    // 1. 初始化一个长度和容量都为 3 的切片
    s1 := make([]int, 3, 3)
    s1[0], s1[1], s1[2] = 1, 2, 3

    // 2. 基于 s1 创建 s2 (共享底层数组)
    s2 := s1[0:2] 
    
    fmt.Printf("修改前 - s1: %v, s2: %v\n", s1, s2)

    // 3. 修改 s2 的元素，s1 也会变（因为它们看的是同一个数组）
    s2[0] = 99
    fmt.Printf("修改后 - s1: %v, s2: %v (同步变化)\n", s1, s2)

    // --- 陷阱触发点 ---

    // 4. 对 s2 进行 append 操作
    // 此时 s2 的 len 是 2，cap 是 3，还有空间，不会扩容
    s2 = append(s2, 88)
    fmt.Printf("Append后(未扩容) - s1: %v, s2: %v (s1 被悄悄改了!)\n", s1, s2)

    // 5. 再次对 s2 进行 append，触发扩容
    // 此时 s2 超过了原容量 3，Go 会为 s2 开辟一块新内存
    s2 = append(s2, 77)
    
    // 6. 再次修改 s2
    s2[0] = 11
    fmt.Printf("扩容后修改 - s1: %v, s2: %v (从此分道扬镳)\n", s1, s2)
}
```

如果希望复制，应该显示调用`copy` :

```go
s2 := make([]int, len(s1))
copy(s2, s1)
```

### 函数传参

- **切片传递是值传递**，传递的是包含指针、长度、容量的结构体副本。
- **共享底层数组**：在不触发扩容的前提下，函数内对元素的修改对外部可见。
- **扩容隔离**：一旦触发扩容，函数内的切片会指向新的内存地址，此后的任何修改（包括 `append` 和元素赋值）对外部切片均不可见。
- **长度隔离**：无论是否扩容，函数内对 `len` 的改变（如 `append`）永远不会反映到外部切片上，除非传递切片指针。

- 如果在函数里传递切片并进行了 `append`，外面的切片会变吗？
    - 如果 `append` **没有**触发扩容，外面的切片**长度不变**（因为长度存放在切片头结构体中，是值传递），但对应的**底层数组数据可能会被改变**。
    - 如果 `append` **触发**了扩容，函数内的切片指向了新内存，对它的任何修改都不会影响外面的切片。

为了避免这种不确定性，Go 社区通常遵循以下原则：

1. **如果你要修改并返回新切片**：函数的签名应该设计为 `func modify(s []int) []int`，调用者使用 `s = modify(s)` 接收。
2. **如果你只是想共享数据**：且确定不需要改变切片的长度（即不 append），直接传切片即可。
3. **如果你非要改变外部切片的长度**：传递**切片指针** `func modify(s *[]int)`，这样修改其 `len` 属性才会对外部生效。

## **Map（映射）**：

• Map 是否线程安全？（不安全。并发读写会 panic，需使用 `sync.Map` 或 `Mutex`）。

### 底层结构

底层是`hmap` 结构体，通过 `buckets` 桶存储，哈希冲突采用链地址法

- **hmap**: 包含 count（元素个数）、B（桶的对数，$2^B$ 个桶）、buckets（指向桶数组的指针）。
- **bmap /bucket**: 每个桶最多存 8 个键值对。为了优化对齐，它是按 `k1,k2...v1,v2...` 的顺序存储，而不是 `k1,v1,k2,v2`。

### 扩容过程

**增量扩容，而不是一次完成。**

1. **翻倍扩容：**装载因子（load factor）> 6.5 时触发。
2. **等量扩容**：溢出桶（overflow buckets）过多时触发，目的是整理碎片。
3. **搬迁**：在每次 `assign`（写）或 `delete`（删）时，顺带搬迁 1-2 个桶的数据到新位置。

### 遍历

Map遍历无序，**随机起始偏移量**。

## **Channel（通道）**：

### 底层结构

`hchan`

- **buf**: 指向循环队列的指针（仅限有缓冲通道）。
- **sendx / recvx**: 记录发送和接收在循环队列中的下标。
- **recvq / sendq**: 阻塞的协程队列，存放的是 `sudog` 结构（封装了 G 和相关数据）。
- **lock**: 互斥锁。**Channel 的底层操作是带锁的**。

### 阻塞流程

- **发送阻塞**：当缓冲区满或无接收者时，当前的 G 会封装成 `sudog` 挂在 `sendq` 队列中，并调用 `gopark` 让出 CPU。
- **接收阻塞**：同理，挂在 `recvq` 队列中等待唤醒。

## Interface

接口分为两种，取决于它是否包含方法：

1. **eface** (empty interface): 不带方法的接口（如 `interface{}`）。包含指向类型的指针和指向数据的指针。
2. **iface**: 带方法的接口。包含一个 `itab`，里面存储了接口类型、具体类型以及**方法表**（用于动态派发）。

**类型断言**：本质上是检查 `itab` 或 `_type` 指针是否匹配。Go 会在编译期或运行期检查具体类型是否实现了接口定义的所有方法。

## 总结

| 特性 | 底层关键点 | **线程安全** | 常见异常 |
| --- | --- | --- | --- |
| Slice | SliceHeader | **否** | 索引越界 |
| Map | hmap, bmap | **否** | 并发读写触发 panic |
| Channel | hchan, sudog | **是** | 向已关闭 channel 写数据触发 panic |
| Interface | iface, eface | **取决于具体数据** | 断言失败（不带 ok 的语法） |

# 并发模型GMP和Goroutine

https://www.youtube.com/watch?v=2oXBOaVowYY

- **GMP 模型详解**：
    - **G (Goroutine)**：协程，2KB 栈起步。
    - **M (Machine)**：工作线程，对应系统线程。
    - **P (Processor)**：处理器，包含运行 G 的上下文和本地队列（核心是减少锁竞争）。
- **调度策略**：
    - 什么是 **Work Stealing**（工作窃取）？（当本地队列为空，从其他 P 偷取一半 G）。
    - 什么是 **Hand Off**（分离机制）？（当 M 因系统调用阻塞，P 会寻找新的 M 绑定）。
- **并发控制**：
    - 如何控制多个协程？（使用 `sync.WaitGroup`、`Context` 或 `Channel`）。

# 垃圾回收

- **GC 垃圾回收**：
    - Go 采用什么 GC 算法？（**三色标记法** + **混合写屏障**）。
    - **STW (Stop The World)** 是什么？在什么时候发生？（主要在标记开始和结束阶段，混合写屏障已极大缩短了 STW 时间）。
- **内存分配**：
    - **逃逸分析 (Escape Analysis)** 是什么？（编译器决定对象分配在栈上还是堆上。若函数返回局部变量指针，则逃逸到堆）。
    - 为什么要减少堆内存分配？（减少 GC 压力，提升性能）。
