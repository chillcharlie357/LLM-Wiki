## Kafka

### Kafka为什么高性能

 Kafka高性能 = **顺序写** + **Page Cache** + **零拷贝** + **批量压缩** + **分区并行**

#### 1. 顺序读写 (Sequential I/O)

Kafka 将消息以 **Append-only** 的方式追加到日志文件（Log）末尾。

- **物理特性**：在磁盘上，顺序 I/O 的速度远高于随机 I/O，甚至可以媲美内存的随机访问速度。它避免了磁盘机械臂频繁寻道（Seek）和旋转延迟。

#### 2. 零拷贝技术 (Zero-Copy)

传统的 I/O 需要在内核缓冲区和用户空间缓冲区之间进行多次数据拷贝（通常是 4 次）。

- **优化**：Kafka 利用操作系统的 `sendfile` 系统调用。数据直接从**页缓存（Page Cache）**传输到 **NIC（网络适配器）**缓冲区，完全跳过了用户态空间，极大地降低了 CPU 切换和内存复制的开销。
- 在 Kafka 的场景下，给 NIC 发送数据是为了完成**从 Broker 到 Consumer（消费者）**的数据传递。

**传统IO：**在传统的 `read` + `write` 模型下，数据经历了 **4 次拷贝** 和 **4 次上下文切换**：

1. **磁盘 -> 内核缓冲区**（Read Buffer）：通过 DMA 拷贝。
2. **内核缓冲区 -> 用户缓冲区**（Application Buffer）：**CPU 拷贝**。此时从内核态切换到用户态。
3. **用户缓冲区 -> Socket 缓冲区**：**CPU 拷贝**。此时从用户态切换回内核态。
4. **Socket 缓冲区 -> 网卡（NIC）**：通过 DMA 拷贝。

**零拷贝**：`sendfile` 系统调用。它直接在内核态完成从“磁盘缓冲区”到“Socket 缓冲区”的传输。

- **磁盘** $\rightarrow$ **内核页缓存 (Page Cache)**：数据被预读进内存。
- **内核页缓存** $\rightarrow$ **NIC 缓冲区**：这是关键的一步。

#### 3. 页缓存 (Page Cache)

Kafka 并不在 JVM 内存中维护庞大的缓存，而是直接利用操作系统的 **Page Cache**。

- **优势**：

- **减少 GC**：由于数据不在 JVM 堆内，避免了频繁的垃圾回收。

- **热数据留存**：即使 Broker 重启，只要系统内核未重启，页缓存依然存在，能够实现“热启动”。

#### 4. 批量处理 (Batching)

Kafka 在 Producer 端和 Consumer 端都采用了批量处理机制。

- **生产者**：攒够一定数量的消息后一次性发送。
- **消费者**：一次拉取（Fetch）一批消息。
这显著降低了网络请求次数和 TCP 封包/拆包的 CPU 消耗。

#### 5. 分区与并发 (Partitioning)

Kafka 的 Topic 被划分为多个 **Partition**。

- **横向扩展**：每个 Partition 只能由同一个 Consumer Group 中的一个 Consumer 处理，但多个 Partition 可以实现真正的并行消费。这种架构消除了单点竞争。

### Kafka vs RabbitMQ vs Redis Stream

| **特性** | **Kafka** | **RabbitMQ** | **Redis Stream** |
| --- | --- | --- | --- |
| **设计核心** | **分布式日志存储** | **消息代理（Exchange/Queue）** | **内存数据结构（Stream）** |
| **存储模型** | 磁盘持久化顺序日志 | 队列模型（通常处理完即删） | 内存存储（可选持久化到磁盘） |
| **数据消费** | **Pull（拉）模式** | **Push（推）模式** | **Pull（拉）模式** |
| **吞吐量** | **极高**（千万级/秒） | 中等（万级/秒） | 高（万级到十万级/秒） |
| **延迟** | 毫秒级 | **微秒级**（更实时） | **微秒级** |
| **数据顺序** | 分区内有序 | 队列内有序 | 消息 ID 递增有序 |
| **典型场景** | 日志采集、大数据链路、流计算 | 金融交易、解耦、复杂路由逻辑 | 轻量级异步任务、实时消息通知 |
